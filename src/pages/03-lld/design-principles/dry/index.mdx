# DRY (Don't Repeat Yourself)

## Introduction
DRY (Don't Repeat Yourself) is arguably the most famous and fundamental principle in software engineering. Formulated by Andy Hunt and Dave Thomas in their book *The Pragmatic Programmer*, it is a guiding rule to reduce repetition of information of all kinds.

## Problem Statement
Imagine writing a web application that needs to connect to a database. You write the connection string and authentication logic in the `UserController`, the `ProductController`, and the `OrderController`. 
Six months later, the database password changes. You update it in `UserController` and `ProductController`, but you forget `OrderController`. Suddenly, checkout breaks in production. This is the danger of WET (Write Everything Twice) code.

## Why this exists
To make code more maintainable, reliable, and readable. When logic is duplicated, maintaining synchronization across all copies becomes an impossible task as a codebase scales. Fixing a bug in one place doesn't fix it everywhere.

## Real-world analogy
Imagine a company's Employee Handbook. If the policy on "Paid Time Off" is written in Chapter 1, Chapter 5, and Chapter 10, updating the policy requires hunting down all three sections. If you miss one, employees get conflicting information. 
The DRY approach is to put the policy in Chapter 1, and in Chapters 5 and 10, simply write: *"Refer to the PTO policy in Chapter 1."*

## Definition
**"Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."**

## Key concepts
- **WET Code:** The opposite of DRY. Stands for "Write Everything Twice" or "We Enjoy Typing."
- **Rule of Three:** A pragmatic guideline for refactoring. If you write the same code twice, let it be. If you find yourself writing it a *third* time, you must refactor it into a shared function.
- **Knowledge vs. Text:** DRY isn't just about literally copy-pasting text. It's about duplicating *knowledge* or *business logic*.

## Python/Java implementation

### Bad Implementation (WET)
```java
class ReportGenerator {
    public void printFinancialReport() {
        System.out.println("--- HEADER ---");
        System.out.println("Date: " + LocalDate.now());
        System.out.println("Generated by: Admin");
        
        System.out.println("Financial Data...");
        
        System.out.println("--- FOOTER ---");
    }

    public void printInventoryReport() {
        // DUPLICATED KNOWLEDGE!
        System.out.println("--- HEADER ---");
        System.out.println("Date: " + LocalDate.now());
        System.out.println("Generated by: Admin");
        
        System.out.println("Inventory Data...");
        
        System.out.println("--- FOOTER ---");
    }
}
```

### Better Implementation (DRY)
```java
class ReportGenerator {
    
    // The authoritative representation of a Header
    private void printHeader() {
        System.out.println("--- HEADER ---");
        System.out.println("Date: " + LocalDate.now());
        System.out.println("Generated by: Admin");
    }

    // The authoritative representation of a Footer
    private void printFooter() {
        System.out.println("--- FOOTER ---");
    }

    public void printFinancialReport() {
        printHeader();
        System.out.println("Financial Data...");
        printFooter();
    }

    public void printInventoryReport() {
        printHeader();
        System.out.println("Inventory Data...");
        printFooter();
    }
}
```

## Multiple real-world examples
1. **Magic Strings/Numbers:** Using the string `"https://api.stripe.com/v1/"` in 10 different files. DRY principle dictates putting this in a central `Config` file or a constant `STRIPE_API_BASE_URL`.
2. **Database Schemas vs Models:** If your database schema defines a `VARCHAR(50)` for a username, and your Java code has validation `if (username.length() > 50)`, that knowledge is duplicated. If the DB changes to `100`, the Java code breaks. Advanced ORMs use the DB schema as the single source of truth to avoid this DRY violation.
3. **CSS Variables:** Instead of writing `color: #3b82f6;` across 50 different CSS classes, you define a variable `--primary-blue: #3b82f6;` and use it. If the brand color changes, you change it in one place.

## Pros
- **Maintainability:** Fix a bug once, and it's fixed everywhere. Update a requirement once, and it propagates globally.
- **Readability:** Extracting duplicated code into well-named functions makes the high-level logic easier to read.
- **Reduces Cognitive Load:** You don't have to remember "every place where the tax calculation is done."

## Cons
- **Premature Abstraction:** This is the biggest risk of DRY. Sometimes two pieces of code *look* identical right now, but they represent entirely different business concepts that will evolve differently (e.g., a User's ID validation and a Product's ID validation both currently checking for `length == 10`). Forcing them into a shared `validateId()` function couples two unrelated concepts. When the User ID rules change later, you break the Product logic.

## Interview questions

### Beginner
- **Q: What does DRY stand for?**
  - **A:** Don't Repeat Yourself.

### Intermediate
- **Q: What is the "Rule of Three"?**
  - **A:** It's a heuristic for knowing when to apply DRY. Writing code twice is acceptable (sometimes abstraction adds too much complexity). Writing it three times is the trigger to refactor it into a shared function.

### Senior
- **Q: Give an example of when violating DRY is actually the correct architectural choice.**
  - **A:** Microservices. In a microservices architecture, you intentionally accept data and code duplication (violating DRY) to achieve strict decoupling. Sharing a `User` class library across 10 microservices creates a tight coupling that defeats the purpose of microservices. It is better for each service to maintain its own slightly duplicated version of the User model.

## Common mistakes
- **Coupling coincidental duplication:** Extracting identical code that just happens to be identical by coincidence, rather than because they represent the same underlying knowledge. (See "Premature Abstraction" above).

## Best practices
- Apply DRY to *business logic* and *knowledge*, not necessarily every single line of identical syntax.
- If abstracting duplicated code makes the code harder to read or requires passing 8 different boolean flags into a master function, it's better to leave it WET. (AHA: Avoid Hasty Abstractions).

## Summary
DRY is the cornerstone of maintainable software. By ensuring that every piece of system knowledge exists in only one authoritative place, developers protect themselves from the cascading bugs and maintenance nightmares that plague heavily duplicated codebases.
