# DRY (Don't Repeat Yourself)

## Introduction
DRY (Don't Repeat Yourself) is a fundamental principle in software engineering. Formulated by Andy Hunt and Dave Thomas in *The Pragmatic Programmer*, it guides developers to minimize repetition of information and logic of all kinds across a system.

## Problem Statement
Imagine writing an application where user phone numbers are validated, normalized, and saved to a database. You write the regex validation and country-code prefix formatting directly in the `CustomerController`, `DriverController`, and `PartnerController`. When the validation rules change (e.g., supporting international numbers), you must locate and update the logic in all three controllers. Forgetting even one controller introduces bugs and data inconsistencies.

## Why this exists
To build maintainable, reliable, and readable software. When business rules or knowledge are duplicated, maintaining sync across all instances becomes difficult as the codebase grows. Fixing a bug or updating a rule in one place fails to propagate to duplicate code paths.

## Real-world analogy
Consider a **company employee handbook**.
If the company's "Paid Time Off" policy is written out in Chapter 1, Chapter 5, and Chapter 10, updating the policy requires editing all three sections. If you miss one, employees receive conflicting rules. The DRY approach is to write the policy in Chapter 1, and reference it in Chapters 5 and 10 with: *"Refer to the PTO policy in Chapter 1."*

Another analogy is **CSS variables**. Instead of typing `#3b82f6` in 50 different stylesheet rules, you define `--primary-blue: #3b82f6` in one root variable. If your company rebrands, you update the color in one place rather than editing 50 lines of CSS.

## Definition
The DRY principle states: Every piece of knowledge must have a single, unambiguous, authoritative representation within a system.
- **Knowledge:** Business logic, algorithms, schemas, or configurations.
- **Authoritative representation:** A single source of truth that defines the behavior.

## Key concepts
- **WET Code (Write Everything Twice):** The opposite of DRY, leading to copy-pasting of logic.
- **Rule of Three:** A refactoring guideline stating that writing code twice is acceptable, but writing it a third time requires refactoring it into a shared component.
- **Duplication of Knowledge vs. Text:** DRY focuses on eliminating duplication of *business rules*, not necessarily identical lines of code that serve different purposes (coincidental duplication).

## Python/Java implementation

### Bad implementation
*Raw validation and formatting logic copy-pasted across three separate controller classes. Any changes to validation rules require editing all three files.*

```java
package bad;

class CustomerController {
    public void registerCustomer(String name, String rawPhone) {
        // Validation: Must be 10 digits
        if (rawPhone == null || !rawPhone.matches("\\d{10}")) {
            throw new IllegalArgumentException("Invalid phone number format");
        }
        String formattedPhone = "+1-" + rawPhone; // Hardcoded US formatting
        System.out.println("Registering Customer " + name + " with " + formattedPhone);
    }
}

class DriverController {
    public void registerDriver(String name, String rawPhone) {
        // Duplicated validation logic!
        if (rawPhone == null || !rawPhone.matches("\\d{10}")) {
            throw new IllegalArgumentException("Invalid phone number format");
        }
        String formattedPhone = "+1-" + rawPhone;
        System.out.println("Registering Driver " + name + " with " + formattedPhone);
    }
}
```

### Better implementation
*Extracting the validation and formatting logic into static methods in a utility helper class. While this reduces repetition, it still exposes raw string variables, failing to prevent invalid string data from being passed elsewhere in the system.*

```java
package better;

class PhoneUtils {
    public static boolean isValid(String phone) {
        return phone != null && phone.matches("\\d{10}");
    }

    public static String formatUS(String phone) {
        return "+1-" + phone;
    }
}

class CustomerController {
    public void registerCustomer(String name, String rawPhone) {
        if (!PhoneUtils.isValid(rawPhone)) {
            throw new IllegalArgumentException("Invalid phone number");
        }
        String formatted = PhoneUtils.formatUS(rawPhone);
        System.out.println("Registering Customer " + name + " with " + formatted);
    }
}
```

### Best implementation
*Encapsulating the logic into a reusable `PhoneNumber` Value Object. The object validates its state upon construction, ensuring that only valid, normalized phone number objects can exist in the domain.*

```java
package best;

import java.util.Objects;

// 1. Authoritative Value Object representing a Phone Number
public final class PhoneNumber {
    private final String formattedNumber;

    public PhoneNumber(String rawNumber) {
        Objects.requireNonNull(rawNumber, "Phone number cannot be null");
        String sanitized = rawNumber.replaceAll("\\D", ""); // Remove non-digits
        
        if (!sanitized.matches("\\d{10}")) {
            throw new IllegalArgumentException("Phone number must contain exactly 10 digits");
        }
        
        // Encapsulate formatting logic
        this.formattedNumber = "+1-" + sanitized;
    }

    public String getFormattedNumber() {
        return formattedNumber;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        PhoneNumber that = (PhoneNumber) o;
        return Objects.equals(formattedNumber, that.formattedNumber);
    }

    @Override
    public int hashCode() {
        return Objects.hash(formattedNumber);
    }
}

// 2. DRY Client classes: depend entirely on the Value Object contract
class CustomerController {
    public void registerCustomer(String name, PhoneNumber phone) {
        // Guaranteed to be valid and formatted! No validation logic duplicated here.
        System.out.println("Registering Customer " + name + " with " + phone.getFormattedNumber());
    }
}

class DriverController {
    public void registerDriver(String name, PhoneNumber phone) {
        System.out.println("Registering Driver " + name + " with " + phone.getFormattedNumber());
    }
}
```

## Step-by-step explanation
1. **Identify Duplication of Knowledge:** We analyze the bad design and find that the regex validation `\\d{10}` and prefix `+1-` are repeated across multiple controllers.
2. **Extract to Value Object:** We create the `PhoneNumber` class. The constructor sanitizes, validates, and formats the input, serving as the single source of truth for phone numbers.
3. **Refactor Client Signatures:** We update the controllers to accept the `PhoneNumber` type instead of a raw `String`. This ensures that only valid, pre-formatted phone numbers can be processed by the application.

## Multiple real-world examples
- **Database Schema Validation:** Modern ORMs (like Hibernate) generate database columns based on annotations in Java entities, ensuring the schema definition exists in only one place.
- **REST Client Configurations:** An application uses a central configuration file to store API base URLs (e.g., `https://api.stripe.com/v1/`), preventing developers from hardcoding the URL across different API client classes.
- **Application Error Constants:** Using an enum or property file (e.g., `ErrorCodes.USER_NOT_FOUND`) to store error messages, ensuring standard messaging across the application.

## Pros
- **Simplified Maintenance:** Updating a business rule requires editing only one class, and the changes propagate globally.
- **Consistent Data:** Validations are enforced uniformly, reducing data corruption.
- **Cleaner APIs:** Reusable objects and methods replace duplicate boilerplates, making the codebase easier to read.

## Cons
- **Risk of Premature Abstraction:** Developers can over-engineer code by abstracting identical logic that just happens to be duplicate by coincidence, introducing unnecessary coupling.

## Interview questions

### Beginner
- **Q: What is the DRY principle?**
- **A:** DRY stands for Don't Repeat Yourself. It states that every piece of system knowledge should have a single, unambiguous representation within the codebase.

### Intermediate
- **Q: What is the difference between coincidental duplication and functional duplication?**
- **A:**
  - **Coincidental Duplication:** Two code segments look identical but serve different business concepts (e.g., validating a User ID and a Product ID both happen to require 10 digits).
  - **Functional Duplication:** Two code segments represent the exact same business rule (e.g., checking a tax rate). DRY should only be applied to functional duplication; abstracting coincidental duplication couples unrelated concepts together.

### Senior
- **Q: What is premature abstraction, and how does it relate to the DRY principle?**
- **A:** Premature abstraction occurs when a developer abstracts code to satisfy DRY before understanding the domain's long-term behavior. This leads to complex, parameterized shared methods with many boolean flags. It is often better to duplicate code temporarily (Write Everything Twice) until patterns emerge.

### Staff Engineer
- **Q: How do you balance the DRY principle with the requirement for service independence in Microservices architectures?**
- **A:** In Microservices, team and service independence takes precedence over the DRY principle:
  1. **Accept Local Duplication:** Avoid creating shared utility libraries (like `common-utils.jar`) containing domain models or logic shared across multiple microservices. This couples the services together, causing updates to one model to break other services.
  2. **Isolate Domain Models:** Each microservice should define and maintain its own domain models, even if they look similar to models in other services.
  3. **Duplicate Infrastructure Details:** Duplicating simple setups (like helper classes or basic configurations) is preferred to preserve decoupling.

## Common mistakes
- **Abstracting coincidental duplication:** Merging two similar code paths that serve different business requirements.
- **Using boolean flags to force reuse:** Writing a massive, complex shared function with multiple flags to handle slightly different use cases, making the code hard to read.

## Best practices
- Apply DRY to *business rules* rather than identical lines of syntax.
- Follow the "Rule of Three" before abstracting duplicate code.
- Keep shared classes small and focused on a single responsibility.

## When NOT to use
- **Decoupled Systems (Microservices):** Avoid sharing domain libraries across microservices to maintain service independence.
- **Test Code:** Test scripts should prioritize readability over strict DRY adherence; using complex helper abstractions can make tests hard to follow.

## Comparison with similar concepts
- **DRY vs AHA (Avoid Hasty Abstractions):**
  - **DRY:** Recommends abstracting duplicated knowledge to prevent bugs.
  - **AHA:** Cautions against abstracting code too early, preferring duplication over the wrong abstraction.

## Summary
The DRY principle ensures that each business rule resides in only one place. Encapsulating logic inside domain objects and helper components prevents data inconsistencies and makes code easier to maintain.
