# Single Responsibility Principle (SRP)

## Introduction
The Single Responsibility Principle (SRP) is the "S" in the SOLID design principles. It is perhaps the most fundamental and universally applicable principle in object-oriented design.

## Problem Statement
When writing software, it is tempting to put everything related to a specific entity into a single class. For example, an `Employee` class might calculate pay, generate a PDF report of their performance, and save the employee data to a MySQL database. 
If the PDF library updates its API, you have to modify the `Employee` class. If the database schema changes, you have to modify the `Employee` class. If the tax calculation changes, you modify the `Employee` class. The class becomes massive, fragile, and a nightmare to test.

## Why this exists
To make classes robust, easier to test, and easier to maintain. By restricting a class to a single responsibility, you isolate changes. When the PDF generation logic breaks, you know exactly which small class to fix, and you know you aren't accidentally breaking the tax calculation logic in the process.

## Real-world analogy
Think of a restaurant. 
- The **Chef** cooks the food.
- The **Waiter** takes the order.
- The **Cashier** handles the payment.
If one person tries to do all three simultaneously (a "God Class"), the restaurant will be chaotic and inefficient. If the credit card machine breaks, the Chef shouldn't have to stop cooking to fix it. Each entity has a Single Responsibility.

## Definition
**"A class should have one, and only one, reason to change."** — Robert C. Martin (Uncle Bob).

## Key concepts
- **Cohesion:** SRP is highly related to cohesion. A class with a single responsibility has high cohesion—its methods and fields are tightly related and focused on one specific goal.
- **God Class:** The anti-pattern of SRP. A class that knows too much or does too much.
- **Separation of Concerns (SoC):** A broader architectural concept that SRP falls under.

## Python/Java implementation

### Bad Implementation (Violating SRP)
```java
// This class has THREE reasons to change:
// 1. Employee business logic changes.
// 2. Database logic changes.
// 3. Report formatting changes.
class Employee {
    private String name;
    private double salary;

    public Employee(String name, double salary) {
        this.name = name;
        this.salary = salary;
    }

    // Responsibility 1: Business Logic
    public double calculatePay() {
        return salary * 0.8; // after taxes
    }

    // Responsibility 2: Database Management
    public void saveToDatabase() {
        System.out.println("Connecting to DB and saving employee...");
    }

    // Responsibility 3: Formatting / Presentation
    public String generateReport() {
        return "<html><body>Employee: " + name + "</body></html>";
    }
}
```

### Better Implementation (Following SRP)
```java
// Reason to change: Only if employee data/logic changes
class Employee {
    private String name;
    private double salary;

    public Employee(String name, double salary) {
        this.name = name;
        this.salary = salary;
    }

    public String getName() { return name; }
    public double getSalary() { return salary; }
}

// Reason to change: Only if tax calculations change
class PayCalculator {
    public double calculatePay(Employee employee) {
        return employee.getSalary() * 0.8;
    }
}

// Reason to change: Only if database type or schema changes
class EmployeeRepository {
    public void save(Employee employee) {
        System.out.println("Saving " + employee.getName() + " to DB...");
    }
}

// Reason to change: Only if the report format (HTML/PDF) changes
class EmployeeReportGenerator {
    public String generateHtmlReport(Employee employee) {
        return "<html><body>Employee: " + employee.getName() + "</body></html>";
    }
}
```

## Multiple real-world examples
1. **User Authentication:** A `User` class holds the username and password hash. A separate `AuthService` handles the actual login logic, token generation, and password validation. A separate `UserRepository` handles saving the user to the DB.
2. **File Processing:** A `FileReader` reads raw bytes from disk. A `FileParser` converts those bytes into JSON objects. A `DataProcessor` analyzes the JSON.
3. **UI Components:** In React/Frontend, a "Container" component fetches data from the API (responsibility 1), and passes that data to a "Presentational" component which only handles rendering the HTML/CSS (responsibility 2).

## Pros
- **Testability:** Testing a `PayCalculator` that just does math is trivial. Testing an `Employee` that connects to a database and formats HTML requires complex mocks.
- **Maintainability:** Bugs are isolated.
- **Reusability:** The `EmployeeReportGenerator` might be reusable for other types of reports, whereas the logic trapped inside the massive `Employee` class was not.

## Cons
- **Class Proliferation:** Following SRP rigidly results in a large number of very small classes. This can sometimes make it harder to trace the "big picture" execution flow of the application.

## Interview questions

### Beginner
- **Q: What is the Single Responsibility Principle?**
  - **A:** A class should have only one reason to change, meaning it should only have one job or responsibility.

### Intermediate
- **Q: How do you know if a class is violating SRP?**
  - **A:** Look at the class imports. If a `Book` class is importing SQL database drivers and HTML rendering libraries, it's doing too much. Also, try describing what the class does. If you use the word "AND" (e.g., "It calculates pay AND saves to the database"), it violates SRP.

### Senior
- **Q: Does SRP mean a class can only have one method?**
  - **A:** No. A `StringManipulation` class can have 50 methods (`trim()`, `toUpperCase()`, `substring()`), but it still has a *single responsibility*: manipulating strings. As long as all methods align with that single purpose, it adheres to SRP.

## Common mistakes
- **Swiss Army Knife Classes:** Creating `Helper` or `Utils` or `Manager` classes (e.g., `AppManager`). These usually become dumping grounds for random methods that violate SRP.

## Best practices
- Keep classes small and focused.
- If a class name ends in `Manager`, `Processor`, or `Super`, take a hard look at it to ensure it isn't a God Class.

## Summary
The Single Responsibility Principle is the foundation of clean architecture. By ensuring every class has only one reason to change, developers create codebases that are modular, testable, and highly resilient to shifting business requirements.
