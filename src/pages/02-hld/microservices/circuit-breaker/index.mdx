# Circuit Breaker

## Introduction
The **Circuit Breaker** is a software resilience design pattern popularized by Michael Nygard in his landmark book *Release It!*. Designed to prevent cascading failures in distributed systems, a circuit breaker wraps calls to remote microservices, monitors failure rates, and instantly redirects or halts requests ("trips the circuit") when a dependency becomes unhealthy.

---

## Problem Statement
In a microservices architecture, services make synchronous network calls to downstream dependencies.
1.  **Cascading Failures:** If Service C fails or slows down, Service B's thread pool begins to block waiting for timeouts. As threads saturate, Service B exhausts its pool, causing it to fail. In turn, Service A fails, causing a chain reaction that takes down the entire system.
2.  **Thread Exhaustion:** Waiting threads block CPU and memory resources. A single slow dependency can freeze thousands of server threads.
3.  **Worsening Downstream Outages:** Continually hammering a struggling service with retries (a **Retry Storm**) prevents it from recovering.

---

## Why This Exists
Circuit breakers exist to provide **fail-fast capabilities** and isolate faults. By intercepting calls at the caller's side, a circuit breaker instantly rejects requests when a threshold is breached, bypassing network latency. Furthermore, it offers a **Fallback Mechanism**—allowing the application to return degraded, default, or cached data to the user rather than raw timeout errors, preserving the user experience.

---

## Real-world Analogy
Imagine the electrical system in your house:
*   **The Dependency:** An electrical appliance (e.g., a toaster).
*   **The Failure:** The toaster has a short circuit, pulling massive amounts of current.
*   **Without a Circuit Breaker:** The current heats up the copper wiring inside your walls. The wires melt, causing a house fire (Cascading Failure).
*   **With a Circuit Breaker:** The moment the current spikes past a safe limit, the physical switch trips inside the fuse box (Open State), cutting off electricity to that room. The rest of the house keeps its lights on.

---

## Definition
**Circuit Breaker** is a state-machine wrapper placed around remote service calls. It tracks request statistics (successes, failures, timeouts) and transitions between **Closed**, **Open**, and **Half-Open** states to protect caller resources and allow downstream recovery.

---

## Key Concepts

### 1. State Machine Transitions
*   **Closed (Normal):** Requests pass through. The breaker monitors failure rates.
*   **Open (Fault Detected):** Requests fail fast immediately, throwing a `CallNotPermittedException` or executing a fallback. The remote service is not called.
*   **Half-Open (Trial):** After a cooldown period (reset timeout), the breaker allows a limited number of requests to pass.
    *   If *all* test calls succeed, the breaker returns to **Closed**.
    *   If *any* test call fails, the breaker returns to **Open**, resetting the cooldown timer.

### 2. Sliding Window Statistics
Modern libraries (like Resilience4j) calculate failure rates using a **Sliding Window**:
*   **Count-based Sliding Window:** Records the last $N$ requests (e.g., last 100 calls). If 50% failed, the breaker opens.
*   **Time-based Sliding Window:** Records calls in the last $T$ seconds (e.g., last 10 seconds).

### 3. Fallback Execution
When the circuit is open (or a call fails), the breaker can invoke a fallback function:
*   *Static Fallback:* Return a default value (e.g., `{"items": []}`).
*   *Cache Fallback:* Retrieve the last known good value from Redis.
*   *Stub Fallback:* Return empty stub models.

## Java Implementation

The following Java code implements a thread-safe, count-based **Sliding Window Circuit Breaker** complete with state transitions, failure rate calculations, and fallback default recovery.

```java
import java.util.Queue;
import java.util.concurrent.ConcurrentLinkedQueue;
import java.util.concurrent.atomic.AtomicReference;

enum BreakerState { CLOSED, OPEN, HALF_OPEN }

public class CustomCircuitBreaker {
    private final int slidingWindowSize;
    private final double failureRateThreshold; // e.g., 0.50 (50%)
    private final long resetTimeoutMs;

    private final AtomicReference<BreakerState> state = new AtomicReference<>(BreakerState.CLOSED);
    private final Queue<Boolean> window = new ConcurrentLinkedQueue<>(); // true = success, false = failure
    
    private long lastStateTransitionTime = System.currentTimeMillis();

    public CustomCircuitBreaker(int windowSize, double failureRateThreshold, int resetTimeoutSec) {
        this.slidingWindowSize = windowSize;
        this.failureRateThreshold = failureRateThreshold;
        this.resetTimeoutMs = resetTimeoutSec * 1000L;
    }

    public synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        BreakerState currentState = state.get();

        if (currentState == BreakerState.OPEN) {
            if (now - lastStateTransitionTime >= resetTimeoutMs) {
                // Transition to Half-Open to test service recovery
                state.set(BreakerState.HALF_OPEN);
                lastStateTransitionTime = now;
                window.clear(); // Reset metrics for trial
                System.out.println("Circuit Breaker transitioned to HALF_OPEN. Initiating trial...");
                return true;
            }
            return false; // Reject request (Fail-fast)
        }
        return true; // Allow request if CLOSED or HALF_OPEN
    }

    public synchronized void recordSuccess() {
        recordCall(true);
    }

    public synchronized void recordFailure() {
        recordCall(false);
    }

    private void recordCall(boolean isSuccess) {
        window.add(isSuccess);
        if (window.size() > slidingWindowSize) {
            window.poll(); // Evict oldest result to slide window
        }

        BreakerState currentState = state.get();
        if (currentState == BreakerState.CLOSED) {
            checkFailureRateAndTrip();
        } else if (currentState == BreakerState.HALF_OPEN) {
            checkHalfOpenRecovery(isSuccess);
        }
    }

    private void checkFailureRateAndTrip() {
        if (window.size() < slidingWindowSize) return; // Wait until window is full

        long failures = window.stream().filter(success -> !success).count();
        double failureRate = (double) failures / window.size();

        if (failureRate >= failureRateThreshold) {
            state.set(BreakerState.OPEN);
            lastStateTransitionTime = System.currentTimeMillis();
            System.err.println("Circuit Breaker TRIPPED to OPEN. Failure rate: " + (failureRate * 100) + "%");
        }
    }

    private void checkHalfOpenRecovery(boolean lastCallSuccess) {
        // In Half-Open, if the trial call fails, trip immediately.
        if (!lastCallSuccess) {
            state.set(BreakerState.OPEN);
            lastStateTransitionTime = System.currentTimeMillis();
            System.err.println("Trial call failed in HALF_OPEN. Tripping back to OPEN.");
        } else {
            // If the trial window gathers enough successful calls, close the circuit
            long successes = window.stream().filter(success -> success).count();
            if (successes >= 3) { // Require 3 consecutive successes to recover
                state.set(BreakerState.CLOSED);
                window.clear();
                System.out.println("Circuit Breaker recovered. Transitioned to CLOSED.");
            }
        }
    }

    public BreakerState getState() {
        return state.get();
    }
}
```

## Step-by-Step Explanation: The Circuit Trip & Recovery
Using the Custom Circuit Breaker above (Window = 5, Threshold = 50%, Cooldown = 5s):

1.  **Closed State:** Five requests are executed. Requests 1 and 2 succeed (`recordSuccess()`). Requests 3, 4, and 5 fail (`recordFailure()`).
2.  **Tripping:** The sliding window holds `[T, T, F, F, F]`. The failure rate is $3/5 = 60\%$. Since $60\% \ge 50\%$, the breaker transitions to `OPEN`.
3.  **Fail-Fast Phase:** A new request arrives. `allowRequest()` returns `false`. The caller bypasses the network and triggers the fallback method immediately.
4.  **Trial Transition:** After 5 seconds, another request arrives. `allowRequest()` detects the timeout has expired and transitions to `HALF_OPEN`. The window is cleared.
5.  **Re-evaluating Recovery:**
    *   *If the trial call fails:* The breaker immediately transitions back to `OPEN`, resetting the 5-second cooldown timer.
    *   *If 3 trial calls succeed consecutively:* The breaker transitions back to `CLOSED`, returning to normal operations.

## Multiple Real-world Examples

1.  **Resilience4j (Standard Java Resiliency):** The industry standard for JVM microservices. It utilizes a sliding ring buffer (array) to store request outcomes and supports customization of retry, bulkhead, and rate-limiting side-cars.
2.  **Netflix Hystrix (Deprecated):** The library that popularized circuit breakers in microservices. It ran commands inside isolated thread pools (Bulkhead isolation), blocking thread starvation from propagating upstream.
3.  **Envoy Proxy / Istio Service Mesh:** Out-of-process circuit breakers. Envoy (running as a sidecar proxy alongside the container) monitors HTTP error codes. If a container returns `503 Service Unavailable` repeatedly, Envoy stops routing traffic to it, handling resiliency at the platform layer.

## Pros & Cons

### Pros
*   **Fault Isolation:** Prevents a single downstream failure from taking down the entire cluster.
*   **Optimal Thread Management:** Freeing waiting threads prevents thread pool starvation.
*   **Downstream Protection:** Stop-the-world behavior gives failing databases/APIs time to recover without being overloaded.
*   **Graceful Degradation:** Fallbacks allow apps to degrade functionality gracefully (e.g., hiding reviews but allowing checkouts).

### Cons
*   **Configuration Complexity:** Choosing the wrong values (sliding window size, failure rate threshold, reset timeout) can cause healthy services to be blocked.
*   **Hard to Test:** Simulating state transitions in testing environments requires writing mock latency/failure injecting runners.
*   **Logic Overhead:** Developers must write fallback methods for every wrapped remote call, increasing codebase size.

## Interview Questions

### Beginner
*   **Q:** What are the three states of a Circuit Breaker?
*   **A:** 
    *   **Closed:** Normal operation; traffic flows.
    *   **Open:** Downstream service is failing; requests fail fast.
    *   **Half-Open:** Cooldown timeout expired; trial requests test if the service has recovered.

### Intermediate
*   **Q:** How does a circuit breaker prevent thread pool exhaustion?
*   **A:** Without a breaker, when a downstream service blocks, caller threads wait for network timeouts, saturating the calling service's thread pool. A circuit breaker in the `OPEN` state rejects requests immediately (fails fast), preventing threads from blocking and saving system resources.

### Senior
*   **Q:** What is the difference between Thread-Isolation (Bulkhead) and Semaphore-Isolation in circuit breakers? Which is preferred?
*   **A:** 
    *   **Thread Isolation:** Executes the remote call on a separate thread pool. If the target service hangs, only that thread pool is exhausted, protecting the main thread pool. However, it introduces CPU context-switching overhead.
    *   **Semaphore Isolation:** Executes the remote call on the calling thread but limits the number of concurrent executions using a counter (semaphore). If the limit is reached, it rejects calls. It has near-zero overhead but relies on callers having strict timeouts.
    *   *Preference:* Semaphore isolation (e.g., used by Resilience4j) is preferred for high-throughput, low-latency APIs due to low CPU overhead. Thread isolation is preferred for untrusted or legacy code where network timeout controls are unreliable.

### Staff Engineer
*   **Q:** How do you coordinate circuit breaker states across 100 distributed instances of a microservice calling the same downstream dependency?
*   **A:** 
    1.  **Local State (Default):** Let each instance maintain its own local sliding window. If Instance 1 detects failures, it trips locally. This is simple, stateless, and has no network overhead.
    2.  **Distributed State (Redis):** If immediate global tripping is required, sync state to a shared Redis cluster using a heartbeat script. However, this introduces latency and Redis dependency on the request path.
    3.  **Service Mesh / Sidecar Sync:** Let Envoy sidecar proxies collect metrics, and a centralized control plane (like Istio) aggregate failure rates and push routing updates to all proxies, separating traffic management from code.

## Common Mistakes
*   **Setting Timeout Longer than Reset Timeout:** If your network timeout is 15 seconds, but your circuit breaker reset timeout is 10 seconds, the breaker will attempt to test recovery before the original call has even timed out.
*   **No Fallback Strategy:** Tripping the circuit but only throwing raw runtime exceptions, which still crashes the user experience instead of degrading gracefully.
*   **Overloading the Half-Open State:** Letting too many trial requests pass during `HALF_OPEN`. The trial pool should be very small (e.g., 3-5 requests).

---

## Best Practices
*   **Pair with Client Timeouts:** Always configure aggressive network connect/read timeouts alongside the circuit breaker.
*   **Use Progressive Cooldowns:** If the circuit enters `HALF_OPEN` and immediately fails again, double the next `reset_timeout` (exponential backoff).
*   **Provide Value-Added Fallbacks:** Return cached or local default values to maintain page rendering.

---

## When NOT to Use
*   **Internal Synchronous Workflows:** Operations that are critical for data consistency (e.g., transactional accounting ledgers).
*   **Stateless Local Computations:** In-memory calculations that do not depend on networks or remote databases.

---

## Comparison with Similar Concepts

*   **Circuit Breaker vs. Retry:** Retries attempt to salvage transient failures (like network blips) on the current request. Circuit breakers stop all requests to protect the system after repeated failures. (Retries run *inside* the closed circuit breaker; if retries still fail, they feed the breaker's failure counter).
*   **Circuit Breaker vs. Rate Limiter:** Rate limiters protect *our* service from being overloaded by clients. Circuit breakers protect *our* service from being pulled down by *other* dependencies.

---

## Summary
The Circuit Breaker pattern is essential for constructing resilient distributed architectures. By failing fast, wrapping calls in count/time-based sliding windows, and executing fallback defaults during open states, systems isolate errors and prevent cascading failures.
