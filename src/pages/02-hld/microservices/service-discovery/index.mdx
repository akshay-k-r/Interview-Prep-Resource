# Service Discovery

## Introduction
In modern microservices architectures, applications are deployed inside dynamically orchestrated containers (e.g., Kubernetes, Docker Swarm, or AWS ECS). As instances scale up, scale down, or crash, their IP addresses and ports change constantly. **Service Discovery** is the mechanism that enables microservices to locate the physical network addresses (IP and Port) of other services dynamically without hardcoding configurations.

---

## Problem Statement
In traditional monolithic architectures, service locations are static (e.g., pointing to `payment.prod.company.com`). However, in cloud-native microservices:
1.  **Ephemeral Instances:** Auto-scaling groups spin up new container instances with random IPs. Hardcoding these endpoints is impossible.
2.  **Configuration Drift:** Manually updating reverse proxy configuration files (like NGINX upstream blocks) every time a container restarts is slow, brittle, and introduces downtime.
3.  **Black-Box Failures:** If a microservice freezes (e.g., due to a database deadlock) but its server stays online, standard DNS routing will still send traffic to it, causing client request timeouts.

---

## Why This Exists
Service discovery replaces static routing with a dynamic **Service Registry**—a highly available database containing real-time coordinates of all healthy service instances. By coupling registration, automated health checking, and query APIs, service discovery ensures that traffic is dynamically routed *only* to active, healthy nodes, enabling seamless horizontal scaling.

---

## Real-world Analogy
Imagine booking a rideshare driver:
*   **Static Config (No Discovery):** You receive a card with the driver's phone number and home address. If they buy a new car or move houses (IP change), you can no longer contact them.
*   **The Service Registry (The Rideshare App):** 
    1.  When a driver goes online, they log into the app, exposing their current GPS coordinates (Service Registration).
    2.  The app runs heartbeats by pinging the driver's phone every 5 seconds (Health Checks).
    3.  When you want a ride, you search the app (Service Query). The app picks a nearby, active driver (Load Balancing) and connects you (Routing).

---

## Definition
**Service Discovery** is a distributed pattern where service instances register their network locations with a **Service Registry** upon startup, and clients or load balancers query the registry to locate and route traffic to healthy instances.

---

## Key Discovery Patterns

### 1. Client-Side Discovery
The calling service is responsible for querying the registry, choosing an instance, and executing the call.
*   *Flow:* Client queries Registry $\to$ Gets list of IPs $\to$ Runs local load balancing (e.g., Round-Robin) $\to$ Calls Service B.
*   *Example:* Spring Cloud Netflix Ribbon/Eureka.

### 2. Server-Side Discovery
The client calls a router/load balancer with a logical name. The router queries the registry and forwards the request.
*   *Flow:* Client calls Load Balancer $\to$ Load Balancer queries Registry $\to$ Load Balancer routes to Service B.
*   *Example:* AWS Application Load Balancer (ALB) or Kubernetes Service (kube-proxy / CoreDNS).

```
Client-Side: Client ---> [Registry] (Reads IPs) ---> Client Load Balancer ---> [Service Instance]
Server-Side: Client ---> [Load Balancer] ---> [Service Instance]
                                | (Registry Sync)
                            [Registry]
```

### 3. Self-Registration vs. Third-Party Registration
*   **Self-Registration:** The service instance itself calls the registry API on startup to register and sends periodic heartbeats. *Simpler to write but couples code to the registry.*
*   **Third-Party Registration (Registrar):** An external orchestrator (like Kubernetes Kubelet) monitors container lifecycle events and automatically registers/deregisters instances. *Keep code clean and registry-agnostic.*

## Java Implementation

The following Java code implements a thread-safe **Service Registry** featuring client-side round-robin load balancing and a background daemon that automatically evicts stale nodes that fail to send heartbeats.

```java
import java.util.*;
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

class ServiceInstance {
    final String serviceName;
    final String ipAddress;
    final int port;
    long lastHeartbeat;

    public ServiceInstance(String serviceName, String ipAddress, int port) {
        this.serviceName = serviceName;
        this.ipAddress = ipAddress;
        this.port = port;
        this.lastHeartbeat = System.currentTimeMillis();
    }

    public String getSocketAddress() {
        return ipAddress + ":" + port;
    }
}

public class DiscoveryServiceRegistry {
    // Maps ServiceName -> List of Instances
    private final Map<String, List<ServiceInstance>> registry = new ConcurrentHashMap<>();
    private final Map<String, AtomicInteger> rrPointers = new ConcurrentHashMap<>();
    
    private final long ttlMs;
    private final ScheduledExecutorService evictorThread = Executors.newSingleThreadScheduledExecutor();

    public DiscoveryServiceRegistry(int ttlSeconds) {
        this.ttlMs = ttlSeconds * 1000L;
        // Start background cleaner to evict dead nodes every 1 second
        this.evictorThread.scheduleAtFixedRate(this::evictStaleInstances, 1, 1, TimeUnit.SECONDS);
    }

    // ==========================================
    // REGISTRATION & HEARTBEAT API
    // ==========================================
    public synchronized void register(String serviceName, String ip, int port) {
        List<ServiceInstance> instances = registry.computeIfAbsent(serviceName, k -> new CopyOnWriteArrayList<>());
        
        // Check if already registered, update heartbeat if yes
        for (ServiceInstance instance : instances) {
            if (instance.ipAddress.equals(ip) && instance.port == port) {
                instance.lastHeartbeat = System.currentTimeMillis();
                System.out.println("Heartbeat received for: " + serviceName + " at " + instance.getSocketAddress());
                return;
            }
        }

        // New registration
        ServiceInstance newInstance = new ServiceInstance(serviceName, ip, port);
        instances.add(newInstance);
        System.out.println("Registered NEW instance: " + serviceName + " -> " + newInstance.getSocketAddress());
    }

    // ==========================================
    // CLIENT-SIDE DISCOVERY (Round-Robin)
    // ==========================================
    public synchronized String discover(String serviceName) {
        List<ServiceInstance> instances = registry.get(serviceName);
        if (instances == null || instances.isEmpty()) {
            return null;
        }

        // Round-robin load balancer
        AtomicInteger pointer = rrPointers.computeIfAbsent(serviceName, k -> new AtomicInteger(0));
        int index = pointer.getAndIncrement() % instances.size();
        return instances.get(Math.abs(index)).getSocketAddress();
    }

    // Background Cleaner Daemon
    private synchronized void evictStaleInstances() {
        long now = System.currentTimeMillis();
        for (Map.Entry<String, List<ServiceInstance>> entry : registry.entrySet()) {
            List<ServiceInstance> list = entry.getValue();
            Iterator<ServiceInstance> iterator = list.iterator();
            while (iterator.hasNext()) {
                ServiceInstance instance = iterator.next();
                if (now - instance.lastHeartbeat > ttlMs) {
                    System.out.println("Evicting dead instance (TTL Expired): " + entry.getKey() + " -> " + instance.getSocketAddress());
                    iterator.remove();
                }
            }
        }
    }

    public void shutdown() {
        evictorThread.shutdown();
    }
}
```

## Step-by-Step Explanation: The Register-Discover Flow
Using the Java registry implementation above:

1.  **Service Boot:** Three instances of `Order-Service` boot. Each executes:
    `register("Order-Service", "10.0.0.X", 8080)`.
2.  **State Verification:** The `DiscoveryServiceRegistry` stores all three endpoints in a list under the key `"Order-Service"`.
3.  **Continuous Heartbeats:** Every 5 seconds, each `Order-Service` instance makes a `register` call (re-submitting heartbeats) to refresh their `lastHeartbeat` timestamps.
4.  **Client Request:** A `Gateway-Service` needs to call `Order-Service`. It calls `discover("Order-Service")`.
5.  **Round-Robin Balancer:** The registry pointer increments, returning `"10.0.0.1:8080"`. On the next call, it returns `"10.0.0.2:8080"`, distributing traffic.
6.  **Crash Eviction:** Instance 3 crashes. It stops sending heartbeats. The background cleaner daemon runs, notices `now - lastHeartbeat > ttlMs`, and removes `"10.0.0.3:8080"` from the active list, shielding it from future client calls.

## Multiple Real-world Examples

1.  **HashiCorp Consul:** A service mesh and discovery registry. It provides DNS interfaces (`order.service.consul`) and REST HTTP APIs. It uses the Gossip protocol for node health check distribution.
2.  **Kubernetes CoreDNS:** In Kubernetes, when you define a `Service` (e.g., `payment-service`), Kubernetes allocates it a virtual IP called a ClusterIP. CoreDNS registers `payment-service` to map to this ClusterIP. Kube-proxy load-balances the traffic to the underlying healthy pods.
3.  **Netflix Eureka:** A REST-based service registry used in the Spring Cloud ecosystem for client-side discovery and load balancing.

## Pros & Cons

### Pros
*   **Elastic Scalability:** Enables nodes to be added or removed dynamically without configuration updates.
*   **Resilience (Auto-Healing):** Detects offline or frozen nodes via heartbeats and routes traffic away automatically.
*   **Decoupled Topologies:** Services only need to know logical names (e.g., `user-service`), not physical IP addresses.

### Cons
*   **Operational SPOF:** The service registry is the central phonebook; if it crashes, microservices cannot find each other, halting system communication.
*   **Staleness Window:** If a node crashes, there is a delay (until the next heartbeat TTL expires) during which clients might still receive the dead node's IP, causing temporary connection failures.
*   **Network Overhead:** Heartbeat traffic from thousands of containers can generate significant network noise.

## Interview Questions

### Beginner
*   **Q:** What is the difference between static routing and dynamic service discovery?
*   **A:** Static routing relies on hardcoded IP addresses or domain configurations that must be updated manually when servers change. Dynamic service discovery uses a central registry where services register their coordinates on startup, allowing other nodes to discover them dynamically via query APIs.

### Intermediate
*   **Q:** Compare Client-Side and Server-Side Service Discovery.
*   **A:** 
    *   **Client-Side:** The client queries the registry directly for a list of healthy instances, uses its own internal library (like Ribbon) to load balance, and calls the selected instance. It is fast (no extra hop) but couples the client code to the registry.
    *   **Server-Side:** The client calls a load balancer (like AWS ALB or K8s Service) using a logical name. The load balancer queries the registry and forwards the traffic. It keeps the client code registry-agnostic but introduces an extra network hop.

### Senior
*   **Q:** What is the "Stale Registry" problem, and how do you mitigate it in client-side discovery?
*   **A:** The stale registry problem occurs when a service instance crashes immediately after sending a heartbeat. The registry still lists it as healthy until the next TTL cycle (e.g., 30s). Clients querying the registry will get the dead IP and experience connection failures. To mitigate this:
    1.  **Client-Side Retry:** If a client request fails with a connection timeout, the client library should immediately try another IP from its local cached list and report the failed IP back to the registry.
    2.  **Short TTL / Heartbeats:** Use short intervals (e.g., 5s heartbeats, 15s TTL).
    3.  **Active Health Checks:** The registry periodically pings nodes instead of relying on passive heartbeats.

### Staff Engineer
*   **Q:** How would you design a highly available, partition-tolerant Service Registry for a cluster of 10,000 microservice nodes? What data structure and consensus model would you choose?
*   **A:** 
    1.  **Registry Topology (AP vs CP):** Service registries are generally read-heavy and require high availability. A temporary inconsistency (e.g., a node taking 5 seconds to show up in the list) is acceptable, but a registration blackout is not. Therefore, choose an **AP (Available/Partition tolerant)** model (like Netflix Eureka or Consul with Gossip) rather than a strict CP consensus store (like ZooKeeper/Raft) which blocks writes if a network partition isolates the leader.
    2.  **Data Structure:** Map-of-Lists structure in memory (using concurrent data structures like `ConcurrentHashMap` and `CopyOnWriteArrayList`) to support $O(1)$ lookups and concurrent updates.
    3.  **Gossip-driven Sync:** Replicate membership lists asynchronously between registry nodes using a Gossip Protocol, avoiding blocking database syncs.
    4.  **Local Caching:** Force clients to cache the registry lists locally, allowing service-to-service communication to continue even if the registry cluster goes offline temporarily.

## Common Mistakes
*   **No Client-Side Fallback Caching:** Designing clients to query the registry on every single API call, creating massive latency and load. Clients should cache entries locally and update them periodically or via event hooks.
*   **Ignoring Health Check Endpoints:** Configuring basic TCP ping health checks. If a service runs out of database connections but stays online, a TCP ping will succeed, but HTTP requests will fail. Health checks should verify all downstream dependencies (database, disk space) are healthy.
*   **Overloading CP Registries:** Using etcd or ZooKeeper for massive clusters with thousands of microservices sending frequent heartbeats. The consensus overhead can saturate the cluster.

---

## Best Practices
*   **Use Third-Party Registrars:** Delegate registration to orchestrators (like Kubernetes) to keep application code clean.
*   **Locally Cache Discovery Lists:** Ensure clients cache IPs locally to insulate them from registry downtime.
*   **Implement Client-Side Retries:** Automatically fallback to another instance if the chosen node is unreachable.

---

## When NOT to Use
*   **Static/Monolithic Deployments:** Small applications with only 1 or 2 static web servers where DNS or static configs are simpler.
*   **P2P Gossip-Only Systems:** Systems where nodes discover each other completely using gossip protocols (like Cassandra nodes) without needing an external registry.

---

## Comparison with Similar Concepts

*   **Service Discovery vs. DNS:** Standard DNS has high caching latencies (TTL propagation delays). Service discovery registries bypass DNS cache buffers to update coordinates instantly (milliseconds).
*   **Service Discovery vs. API Gateway:** Service discovery resolves internal addresses. An API gateway sits at the public boundary, routing external client traffic using discovery lists internally.

---

## Summary
Service discovery is essential for routing traffic in dynamic, cloud-native microservice deployments. By using a highly available service registry coupled with automated health checking, round-robin load balancing, and client-side fallback caches, systems can scale dynamically and self-heal automatically.
