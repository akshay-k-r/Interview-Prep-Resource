# Event Sourcing

## Introduction
**Event Sourcing** is a software design pattern where all state modifications of an application are captured and persisted as an immutable sequence of chronological events, rather than storing only the current, updated state of records. The event log itself is the definitive **source of truth** from which any past or present application state is reconstructed.

---

## Problem Statement
In traditional CRUD (Create, Read, Update, Delete) databases, executing an update (e.g., changing a user's address from "New York" to "Paris") overwrites the old value in the table. This introduces several critical limitations:
1.  **Loss of History:** The system forgets the previous state. You cannot ask: *"What was this user's address on August 12, 2025?"* without separate, complex audit tables.
2.  **Audit Trail Delays:** Creating audit logs relies on developers manually writing log statements or database triggers, which is error-prone and decoupled from the business transaction.
3.  **Concurrency Conflicts:** Multiple threads performing read-modify-write operations on the same row cause lock contention and transaction aborts.

---

## Why This Exists
Event Sourcing guarantees a 100% accurate, complete history of the application. Because events are written sequentially to an append-only log, writes are extremely fast and deadlock-free. By replaying this event log from index 0, the application can reconstruct its state to **any point in time** (Time-Travel). It is the de-facto standard for financial ledgers, auditing compliance systems, and high-performance event-driven microservices.

---

## Real-world Analogy
Imagine maintaining a personal bank checking account:
*   **Traditional CRUD Database:** You keep a blackboard that only shows your current balance: `$500`. You do not know how you got this money, if you spent $1000 yesterday, or if someone stole your card.
*   **Event Sourcing (The Banking Ledger):** You keep an immutable notebook. You write every transaction:
    1.  `AccountOpened` (Deposited +$1000)
    2.  `CashWithdrawn` (-$300)
    3.  `SalaryCredited` (+$800)
    4.  `UtilitiesPaid` (-$1000)
    *   To find your balance, you calculate the sum: $1000 - 300 + 800 - 1000 = \$500$. The ledger is the source of truth, and the balance is a projection.

---

## Definition
**Event Sourcing** is a data persistence pattern where the state of an application entity (Aggregate) is modeled as a sequence of immutable domain events, stored in an append-only **Event Store**, and reconstructed dynamically on demand.

---

## Key Concepts

### 1. The Event Store
A specialized, append-only database optimized for storing sequential event streams. Popular implementations include EventStoreDB, or custom implementations built on top of relational databases (PostgreSQL) or NoSQL (DynamoDB).

### 2. Snapshots
If an aggregate has 100,000 events, reading and replaying them on every request to find its current state is slow.
*   **The Solution:** Periodically (e.g., every 100 events), the system serializes the aggregate's current state and saves it to a **Snapshot Store**. When loading, the system reads the latest snapshot and replays only the events that occurred *after* that snapshot.

```
Events: [1] ... [99] -> [Snapshot at 100] -> [101] [102]
Loading Path: Read Snapshot 100 + Replay 101 and 102
```

### 3. Event Upcasting
As business requirements change, event schemas evolve. A new version of an event might add fields. Because historical events are immutable and cannot be rewritten on disk, the system uses **Upcasters**—middleware components that intercept old event formats during loading and translate them dynamically to the newest schema before passing them to the aggregate.

## Java Implementation

The following Java code provides a complete, thread-safe implementation of an **Event-Sourced Shopping Cart Aggregate**. It demonstrates event creation, state reconstruction via replay, and snapshot checkpoint optimization.

```java
import java.util.*;
import java.util.concurrent.ConcurrentHashMap;

// --- 1. IMMUTABLE DOMAIN EVENTS ---
interface DomainEvent {
    int getVersion();
}

class CartCreatedEvent implements DomainEvent {
    final String cartId;
    final int version;
    public CartCreatedEvent(String cartId, int version) { this.cartId = cartId; this.version = version; }
    public int getVersion() { return version; }
}

class ItemAddedEvent implements DomainEvent {
    final String item;
    final double price;
    final int version;
    public ItemAddedEvent(String item, double price, int version) { this.item = item; this.price = price; this.version = version; }
    public int getVersion() { return version; }
}

// --- 2. AGGREGATE ROOT (Shopping Cart) ---
class ShoppingCartAggregate {
    private final String id;
    private int version = 0;
    private final List<String> items = new ArrayList<>();
    private double totalPrice = 0.0;
    private boolean isCheckedOut = false;

    public ShoppingCartAggregate(String id) {
        this.id = id;
    }

    public void apply(DomainEvent event) {
        if (event instanceof CartCreatedEvent) {
            this.version = event.getVersion();
        } else if (event instanceof ItemAddedEvent) {
            ItemAddedEvent itemEvent = (ItemAddedEvent) event;
            this.items.add(itemEvent.item);
            this.totalPrice += itemEvent.price;
            this.version = event.getVersion();
        }
    }

    public int getVersion() { return version; }
    public List<String> getItems() { return items; }
    public double getTotalPrice() { return totalPrice; }
}

// --- 3. EVENT STORE & SNAPSHOT ENGINE ---
public class EventSourcedEngine {
    private final Map<String, List<DomainEvent>> eventStore = new ConcurrentHashMap<>();
    private final Map<String, ShoppingCartAggregate> snapshotStore = new ConcurrentHashMap<>();

    public void appendEvent(String cartId, DomainEvent event) {
        eventStore.computeIfAbsent(cartId, k -> new ArrayList<>()).add(event);
        
        // Periodically take snapshots (e.g. every 2 events for demo purposes)
        if (event.getVersion() % 2 == 0) {
            takeSnapshot(cartId);
        }
    }

    private void takeSnapshot(String cartId) {
        System.out.println("Snapshotting Cart: " + cartId);
        ShoppingCartAggregate activeState = load(cartId);
        // Save copy of current state
        ShoppingCartAggregate snapshotCopy = new ShoppingCartAggregate(cartId);
        snapshotCopy.getItems().addAll(activeState.getItems());
        snapshotCopy.apply(new CartCreatedEvent(cartId, activeState.getVersion()));
        snapshotStore.put(cartId, snapshotCopy);
    }

    // =====================================================================
    // STATE RECONSTRUCTION VIA SNAPSHOT + REPLAY
    // =====================================================================
    public ShoppingCartAggregate load(String cartId) {
        ShoppingCartAggregate cart = snapshotStore.get(cartId);
        int startVersion = 0;

        if (cart != null) {
            System.out.println("Loaded snapshot checkpoint for: " + cartId + " at Version: " + cart.getVersion());
            startVersion = cart.getVersion();
        } else {
            cart = new ShoppingCartAggregate(cartId);
        }

        List<DomainEvent> events = eventStore.get(cartId);
        if (events != null) {
            for (DomainEvent event : events) {
                // Replay only events that occurred after the snapshot version
                if (event.getVersion() > startVersion) {
                    System.out.println("Replaying Event: " + event.getClass().getSimpleName() + " (Version: " + event.getVersion() + ")");
                    cart.apply(event);
                }
            }
        }
        return cart;
    }
}
```

## Step-by-Step Explanation: The State Replay Flow
Using the Java Event Sourced Engine above:

1.  **Creation:** An application creates cart `Cart-101`. It appends a `CartCreatedEvent` (Version 1) to `eventStore`.
2.  **Item Appending:** A user adds "Laptop" ($1000). The engine appends `ItemAddedEvent("Laptop", 1000, Version 2)`.
3.  **Snapshot Trigger:** The engine detects version 2 is a multiple of 2, so it invokes `takeSnapshot("Cart-101")` and caches the state copy.
4.  **Further Edits:** The user adds "Mouse" ($50). The engine appends `ItemAddedEvent("Mouse", 50, Version 3)`.
5.  **State Reconstruction Request:** The user views their checkout page. The application calls `load("Cart-101")`.
6.  **Optimized Load:** The engine reads the snapshot at version 2 (containing `"Laptop"`). It sees that the total version history in `eventStore` reaches version 3.
7.  **Selective Replay:** It skips events at version 1 and 2, and replays only `Version 3` (`"Mouse"`), returning the final cart.

## Multiple Real-world Examples

1.  **Git Version Control:** Git does not save your files as monolithic code blobs on every save. Instead, it saves file changes as a sequence of **commits** (events). Replaying commits constructs the current file state. You can revert commits or branch out to see past states (time-travel).
2.  **Financial General Ledgers:** Bank balance databases only append transactions (debit/credit events). Replaying all transactions yields the balance, maintaining an audit trial that is compliant with international financial regulations.
3.  **Collaborative Document Editing (Google Docs):** Uses operational transformations on a stream of keystroke events. Replaying the event logs allows users to see edit history and restore previous versions.

## Pros & Cons

### Pros
*   **Perfect Auditing:** Every state mutation is tracked. There is no silent data corruption.
*   **Time-Travel Capabilities:** Replaying events up to a specific timestamp allows developers to debug production bugs by reconstructing the exact state of the system when the error occurred.
*   **Write Performance:** The event store is append-only, avoiding update locking, and maximizing I/O performance.
*   **Decoupled Rebuilds:** If a read-model database (MongoDB) is corrupted, it can be deleted and reconstructed completely from scratch by replaying the event log.

### Cons
*   **Complexity:** Diverging from CRUD-oriented thinking requires significant engineering learning curves.
*   **Eventual Consistency Lag:** Pairing Event Sourcing with CQRS means read views are eventually consistent, adding UI/UX synchronization complexity.
*   **Schema Evolution Cost:** Managing event version changes (Upcasting) requires writing custom translation classes over the lifecycle of the system.

## Interview Questions

### Beginner
*   **Q:** What is the core difference between CRUD and Event Sourcing?
*   **A:** In CRUD, the database stores only the latest, current state of the records, overwriting old data. In Event Sourcing, the database stores all historical state changes as an immutable sequence of events; current state is derived by replaying these events.

### Intermediate
*   **Q:** What is a Snapshot in Event Sourcing, and why is it used?
*   **A:** A snapshot is a serialized copy of an aggregate's state at a specific version (e.g., after every 100 events). It is used to optimize load performance: instead of replaying thousands of historical events on every read request, the system loads the latest snapshot and replays only the events that occurred after it.

### Senior
*   **Q:** What is Event Upcasting, and how does it handle schema changes in historical event stores?
*   **A:** Since historical event stores are immutable, old events stored on disk cannot be edited when schemas change (e.g., adding a new field). Event Upcasting is a pattern where a translation middleware component intercepts old event versions during the database read path and updates their JSON structure dynamically to match the current schema version before the aggregate processes them.

### Staff Engineer
*   **Q:** Design an Event Sourced ledger system for a bank that processes 10,000 transactions per second. How do you handle concurrency conflicts (e.g., double-spending) in an append-only event store?
*   **A:** 
    1.  **Optimistic Concurrency Control (OCC):** Every event stream (aggregate) maintains an incrementing version number.
    2.  **Atomic Append:** When appending a new event, the write query must include a constraint checking that the current database version matches the expected version:
        `INSERT INTO events (aggregate_id, version, payload) VALUES ('acc_12', 4, '...') WHERE (SELECT MAX(version) FROM events WHERE aggregate_id = 'acc_12') = 3`
    3.  **Conflict Handling:** If two threads attempt to spend money concurrently, both read version 3. The first thread writes version 4 successfully. The second thread's write fails due to a version conflict. The second thread must reload the stream, apply the new events to recalculate the balance, verify that funds are still available, and attempt to write again under the new version.

## Common Mistakes
*   **Using for Simple CRUD Apps:** Implementing Event Sourcing for basic admin panels with low change rates.
*   **No Snapshots:** Forgetting to implement snapshotting, leading to system degradation as log sizes grow.
*   **Mutating Historical Logs:** Attempting to run `UPDATE` statements on the event store, violating the immutability contract.

---

## Best Practices
*   **Keep Events Small:** Ensure events capture a specific, narrow fact (e.g., `EmailUpdated` rather than `UserModified`).
*   **Pair with CQRS:** Use CQRS to project events into flat read-model databases, keeping queries fast and simple.
*   **Implement Upcasters Early:** Design event serialization to support version metadata in the headers from day one.

---

## When NOT to Use
*   **Strict Real-Time Immediate Consistency:** Systems that cannot tolerate any eventual consistency delays in their read paths.
*   **Low-History CRUD Systems:** Standard catalog inventories or static settings dashboards.

---

## Comparison with Similar Concepts

*   **Event Sourcing vs. Event-Driven Architecture:** Event Sourcing is a persistence pattern *internal* to a single service (storing state changes as events). Event-Driven Architecture is an integration pattern for communicating *between* distinct microservices.
*   **Event Sourcing vs. Transactional Outbox:** The outbox pattern writes a state update and an event to separate tables atomically. Event Sourcing writes *only* the event, deriving the state dynamically.

---

## Summary
Event Sourcing treats state changes as immutable, historical records, ensuring perfect auditing and enabling time-travel debugging. By combining append-only event stores with snapshot checkpoints, event upcasting, and CQRS projections, developers can construct resilient, high-performance distributed systems.
