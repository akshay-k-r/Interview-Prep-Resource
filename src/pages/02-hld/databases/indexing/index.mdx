# Indexing

## Introduction
Indexing is a database optimization technique that creates auxiliary data structures to accelerate data retrieval. By avoiding sequential table scans, indexes allow query engines to locate and fetch records in logarithmic or constant time, which is critical for maintaining low latency as datasets scale to millions or billions of rows.

---

## Problem Statement
Without indexes, a database must perform a **Full Table Scan (Sequential Scan)** to evaluate any query predicate (e.g., `WHERE age = 30`). This means reading every block of data from disk into memory, checking each record individually, and discarding non-matching rows. For a table with $N$ records, a sequential scan runs in $O(N)$ time. As $N$ grows, I/O bandwidth and CPU limits are quickly exceeded, leading to system degradation and timeouts.

---

## Why This Exists
In database storage engines, data is organized into fixed-size blocks or pages (e.g., 4KB or 8KB) on disk. Reading from disk is orders of magnitude slower than reading from memory. Indexes exist to:
1. **Minimize Disk I/O:** Directly map search keys to specific data block addresses (offsets) so the engine only reads pages containing target records.
2. **Transform Time Complexity:** Reduce lookup times from linear $O(N)$ to logarithmic $O(\log N)$ or constant $O(1)$.
3. **Optimize Join & Order Operations:** Allow sorting (`ORDER BY`), grouping (`GROUP BY`), and joins (`JOIN`) to execute without expensive in-memory sorts.

---

## Real-world Analogy
Imagine entering a library with 100,000 books. 
- **Without an Index (Sequential Scan):** To find "The Great Gatsby," you would have to start at the first shelf of the first aisle and inspect the cover of every single book until you find it.
- **With a Clustered Index:** The library books are physically arranged alphabetically by title on the shelves. You can binary search the aisles to find the exact shelf.
- **With a Non-Clustered Index:** The books are randomly stacked, but there is a physical card catalog sorted alphabetically by title. Each card contains a reference number (like a shelf/row locator) telling you exactly where the book is located.

---

## Definition
An **Index** is a specialized, persistent data structure that references keys from one or more columns in a table and maps them to the corresponding storage locations of the rows. It traded write latency and storage space for read speed.

---

## Key Concepts

### 1. B-Tree vs. B+ Tree Indexes
*   **B-Tree (Balanced Tree):** Every node (internal and leaf) stores both keys and the actual data records (or pointers to them).
*   **B+ Tree (De-facto Standard for RDBMS):** 
    *   **Internal Nodes:** Only store keys and child pointers to direct the search. This maximizes the **fan-out** (number of children per node), keeping the tree extremely shallow.
    *   **Leaf Nodes:** Store the actual data or pointers to data, and are linked together in a **doubly linked list**. This allows for extremely efficient range queries (e.g., `WHERE age BETWEEN 20 AND 30`) by doing a tree search for the lower bound, then traversing the linked list.

```
                  [ 50 | 100 ]            <-- Internal Node (Keys + Pointers only)
                 /     |      \
        [ 20 | 35 ]  [ 70 ]  [ 120 ]      <-- Internal Node
        /    |   \   /    \  /     \
    [10|15]=>[20]=>[50]=>[80]=>[100]=>[130] <-- Leaf Nodes (Keys + Data Pointers + Linked List pointers)
```

### 2. LSM Tree (Log-Structured Merge-tree)
Optimized for write-heavy workloads (used in NoSQL engines like Cassandra, RocksDB).
*   **MemTable:** In-memory sorted buffer (usually a SkipList or Red-Black Tree) where all writes and updates are buffered.
*   **Write-Ahead Log (WAL):** Sequential log file on disk to guarantee durability before updating MemTable.
*   **SSTables (Sorted String Tables):** Immutable files written to disk when the MemTable fills up.
*   **Compaction:** Background processes that merge SSTables, discarding overwritten/deleted keys (tombstones).

### 3. Clustered vs. Non-Clustered (Secondary) Indexes
*   **Clustered Index:** Defines the physical order in which data is stored on disk. A table can have **only one** clustered index (typically the Primary Key). The leaf nodes of a clustered index contain the actual data rows.
*   **Non-Clustered Index:** A separate structure from the data rows. The leaf nodes contain the index key and a pointer (Row Identifier / RID or the Clustered Index Key) to the actual data row.

### 4. Special Index Types
*   **Hash Index:** Uses a hash function to map keys to bucket pointers. Ideal for exact matches ($O(1)$ lookup) but does not support range scans or sorting.
*   **Inverted Index:** Maps terms (words) to the documents containing them. This is the foundation of full-text search engines like Elasticsearch.
*   **Composite Index:** An index built on multiple columns (e.g., `(last_name, first_name)`). It follows the **Left-Most Prefix Rule**—meaning a query on `last_name` can use the index, but a query on `first_name` alone cannot.
*   **Covering Index:** A non-clustered index that contains all columns requested by a query. The database engine can satisfy the query entirely from the index page without performing a lookup (bookmark lookup) on the data page.

## Java Implementation

The following example demonstrates index search patterns:
1.  **Bad:** Full sequential search.
2.  **Better:** In-memory Hash Index lookup (fast equality, no range query).
3.  **Best:** A basic B+ Tree style node implementation for sorted range scans.

```java
import java.util.*;

// Common record definition
class User {
    int id;
    String name;
    int age;

    public User(int id, String name, int age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }
    @Override
    public String toString() {
        return "User{id=" + id + ", name='" + name + "', age=" + age + "}";
    }
}

// ==========================================
// 1. BAD IMPLEMENTATION: Full Table Scan
// ==========================================
class UnindexedTable {
    private final List<User> rows = new ArrayList<>();

    public void insert(User user) {
        rows.add(user);
    }

    // O(N) lookup
    public List<User> findByAge(int age) {
        List<User> results = new ArrayList<>();
        for (User user : rows) {
            if (user.age == age) {
                results.add(user);
            }
        }
        return results;
    }
}

// ==========================================
// 2. BETTER IMPLEMENTATION: Hash Indexed Table
// ==========================================
class HashIndexedTable {
    private final List<User> rows = new ArrayList<>();
    private final Map<Integer, List<Integer>> ageIndex = new HashMap<>(); // Maps Age -> Row Indexes

    public void insert(User user) {
        int rowIndex = rows.size();
        rows.add(user);
        ageIndex.computeIfAbsent(user.age, k -> new ArrayList<>()).add(rowIndex);
    }

    // O(1) average lookup for equality, but CANNOT do range queries efficiently
    public List<User> findByAge(int age) {
        List<Integer> rowIds = ageIndex.get(age);
        if (rowIds == null) return Collections.emptyList();
        
        List<User> results = new ArrayList<>();
        for (int id : rowIds) {
            results.add(rows.get(id));
        }
        return results;
    }
}

// ==========================================
// 3. BEST IMPLEMENTATION: Tree-Based Index (Supports Range Queries)
// ==========================================
class TreeMapIndexedTable {
    private final List<User> rows = new ArrayList<>();
    // TreeMap uses a Red-Black Tree (Self-balancing Binary Search Tree)
    // Keys are sorted, giving O(log N) lookups and O(log N + M) range queries
    private final TreeMap<Integer, List<Integer>> ageTreeIndex = new TreeMap<>();

    public void insert(User user) {
        int rowIndex = rows.size();
        rows.add(user);
        ageTreeIndex.computeIfAbsent(user.age, k -> new ArrayList<>()).add(rowIndex);
    }

    // O(log N) equality search
    public List<User> findByAge(int age) {
        List<Integer> rowIds = ageTreeIndex.get(age);
        if (rowIds == null) return Collections.emptyList();
        
        List<User> results = new ArrayList<>();
        for (int id : rowIds) {
            results.add(rows.get(id));
        }
        return results;
    }

    // Range query support: O(log N + M) where M is matches
    public List<User> findByAgeRange(int minAge, int maxAge) {
        List<User> results = new ArrayList<>();
        // Get submap representing range
        SortedMap<Integer, List<Integer>> subMap = ageTreeIndex.subMap(minAge, true, maxAge, true);
        
        for (List<Integer> rowIds : subMap.values()) {
            for (int id : rowIds) {
                results.add(rows.get(id));
            }
        }
        return results;
    }
}
```

## Step-by-Step Explanation: B+ Tree Range Lookup
When executing `SELECT * FROM users WHERE age BETWEEN 25 AND 27`:

1.  **Traverse Tree (Root to Leaf):** The engine begins at the B+ tree root. It checks the keys to determine which pointer to follow. For example, if the root has key `20` and `50`, and our target is `25`, it follows the middle pointer pointing to the child node containing keys between `20` and `50`.
2.  **Find Lower Bound Leaf Node:** The engine navigates down the tree levels until it reaches the leaf level, landing on the leaf node containing the key `25` (representing the lower bound of the range).
3.  **Horizontal Scan:** Because leaf nodes are connected via a doubly linked list, the database does not need to traverse back up the tree. It simply reads sequentially along the leaf node list, harvesting records for `25`, `26`, and `27`.
4.  **Terminate:** The scan stops as soon as it encounters a key greater than the upper bound (`27`), saving substantial CPU and disk cycles.

---

## Multiple Real-world Examples

1.  **MySQL (InnoDB):** Implements B+ Trees for both clustered indexes (primary key) and secondary indexes. Primary keys store data in leaf nodes; secondary indexes store the primary key value as a pointer.
2.  **PostgreSQL (GIN Index):** Generalized Inverted Indexes (GIN) are optimized for searching multi-valued data types like JSONB arrays, TSVectors (full-text search), and range types.
3.  **Elasticsearch / Lucene:** Uses inverted indexes consisting of a **Term Dictionary** (stored in a prefix tree / finite state transducer) and **Postings Lists** (sorted arrays of document IDs) for full-text search.
4.  **Apache Cassandra:** Utilizes LSM Trees (with Memtables and SSTables) coupled with Bloom Filters to quickly verify if an SSTable contains a requested key before doing disk read operations.

## Pros & Cons

### Pros
*   **High Read Throughput:** Queries targeting specific keys run in $O(\log N)$ or $O(1)$ rather than $O(N)$.
*   **Ordered Scan Execution:** Eliminates sorting overhead (`ORDER BY`) if the index preserves the required order.
*   **Unique Constraints Enforcement:** Unique indexes prevent duplicate database entries efficiently.
*   **Improves Joins:** Foreign key indexes significantly speed up relation matching during nested-loop joins.

### Cons
*   **Write Penalty:** Every `INSERT`, `UPDATE`, and `DELETE` must update the primary table *and* modify all associated index structures.
*   **Disk Space Overhead:** Indexes consume significant storage space. Large databases can have indexes that exceed the size of the raw data.
*   **Memory Pressure:** To remain fast, indexes should ideally fit entirely in the database cache (Buffer Pool). Large indexes lead to page eviction and disk thrashing.

## Interview Questions

### Beginner
*   **Q:** What is the difference between a Clustered and a Non-Clustered index?
*   **A:** A clustered index determines the physical order of rows on disk and stores the actual data values at the leaf level (limit 1 per table). A non-clustered index is a separate structure containing the indexed column keys and pointers back to the original rows (can have many per table).

### Intermediate
*   **Q:** Why does a B+ Tree index perform better than a standard Binary Search Tree in disk-based databases?
*   **A:** Binary Search Trees have a fan-out of 2, leading to deep trees and multiple disk seeks. B+ Trees have a high fan-out (often $100+$), keeping the height shallow (usually 3 or 4 levels) even for millions of records. This limits disk I/O to a maximum of 3-4 read operations per query.

### Senior
*   **Q:** Explain the Left-most Prefix Rule in composite indexes and how a skip scan works.
*   **A:** A composite index on `(A, B, C)` is sorted hierarchically: first by `A`, then by `B` within `A`, and so on. A query filtering on `(B, C)` cannot easily use the index because the rows are not grouped sequentially by `B` globally. However, if `A` has low cardinality (few distinct values), some engines perform an optimization called an "Index Skip Scan," where they scan the sub-trees for each distinct value of `A` to search for `(B, C)`.

### Staff Engineer
*   **Q:** What is Write Amplification in the context of database indexing, and how do LSM Trees mitigate it compared to B+ Trees?
*   **A:** Write Amplification is the ratio of bytes written to underlying storage relative to the logical bytes written by the user. In B+ Trees, updating a single record forces the database to write the entire page (usually 8KB-16KB) to disk, causing high write amplification. LSM Trees write updates sequentially to a memory buffer (MemTable) first and flush them as large, contiguous, sorted files (SSTables) to disk. This converts random random I/O writes into sequential writes, greatly reducing write amplification for write-intensive workloads.

## Common Mistakes
*   **Indexing Columns of Low Selectivity:** Creating a B+ Tree index on columns like `gender` or `status` (which only have a few distinct values). The query planner will typically ignore the index and perform a table scan anyway.
*   **Failing to Cover Queries:** Querying `SELECT *` when only two columns are needed. If those two columns were indexed, a covering index search would avoid retrieving the entire row from the clustered storage.
*   **Over-Indexing:** Adding indexes for every single column. This degrades insert/update performance and bloats the storage size.

---

## Best Practices
*   **Index Foreign Keys:** Columns used in `JOIN` conditions should almost always be indexed to speed up relations.
*   **Apply the Left-most Rule Correctly:** Design composite indexes so columns filtered with equality constraints (`=`) appear first, and range filters (`>`, `<`) appear last.
*   **Regularly Rebuild/Defragment Indexes:** Over time, page splits lead to index fragmentation. Regular maintenance keeps the tree structure dense and contiguous.

---

## When NOT to Use
*   **Small Tables:** If a table fits within a single disk block, performing a sequential scan is faster than traversing an index and fetching the row.
*   **High-Write, Low-Read Workloads:** If data is rarely read but updated constantly, the index maintenance overhead outweighs the query optimization benefits.

---

## Comparison with Similar Concepts

*   **Indexing vs. Partitioning:** Indexing builds a lookup path *within* a dataset. Partitioning physically splits a large table into smaller sub-tables based on a boundary key (e.g., partitioning by year).
*   **Indexing vs. Caching:** Indexing keeps structured search paths on disk/memory for all data. Caching stores the complete pre-computed query results in memory (e.g., Redis) for specific key hits.

---

## Summary
Indexing is a critical tool for database performance. While B+ Trees remain the de-facto choice for traditional read-heavy relational databases, write-heavy systems rely on LSM Trees. Selecting, designing, and maintaining the correct index pattern is essential for any high-scale software architecture.
