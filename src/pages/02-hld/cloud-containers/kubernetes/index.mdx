# Kubernetes (K8s)

## Introduction
Kubernetes (often abbreviated as K8s) is an open-source container orchestration system originally designed by Google. It automates the deployment, scaling, management, and networking of containerized applications across a cluster of machines.

## Problem Statement
Docker is fantastic for running a few containers on a single machine. But what if your e-commerce site has 500 microservices running across 50 servers? 
- If a server crashes, how do you automatically restart its containers on a healthy server?
- If traffic spikes, how do you automatically spin up more instances of the checkout service?
- How do these hundreds of containers discover and talk to each other securely?
Managing this manually with raw Docker is impossible.

## Why this exists
To provide a highly automated, resilient, and declarative "operating system for the cloud." It ensures that the actual state of your infrastructure perfectly matches your desired state, handling all the scaling, healing, and routing automatically.

## Real-world analogy
If Docker containers are musicians in an orchestra (each playing their instrument perfectly in isolation), Kubernetes is the **Conductor**. The conductor doesn't play the instruments, but they ensure everyone plays together at the right tempo, brings in more violinists if the music needs to be louder, and points to a backup if a musician gets sick.

## Definition
Kubernetes is a portable, extensible, open-source platform for managing containerized workloads and services, that facilitates both declarative configuration and automation.

## Key concepts
- **Cluster:** A set of worker machines, called nodes, that run containerized applications.
- **Node:** A physical or virtual machine in the cluster.
- **Pod:** The smallest deployable computing unit in K8s. It encapsulates one or more tightly coupled containers that share storage and network resources.
- **Control Plane (Master):** The brain of the cluster. It manages the nodes and the pods. Components include the API Server, Scheduler, etcd (key-value store of cluster state), and Controller Manager.
- **Deployment:** A declarative object that tells K8s how to run your pods, including how many replicas to maintain and how to perform rolling updates.
- **Service:** An abstract way to expose an application running on a set of Pods as a stable network service (provides a static IP and load balancing).

## Python/Java implementation

Below is a Java simulation demonstrating how the Kubernetes Control Plane reconciliation loop works under the hood.

### Bad implementation
*A static shell execution loop. If a shell process crashes or the node runs out of memory, there is no reconciliation, self-healing, or capacity tracking, leaving the application permanently offline.*

```java
import java.io.IOException;

// BAD: Static launcher loop. No rescheduling, node health checks, or target counts.
public class FragileRunner {
    public void startAppInstances(int count) {
        for (int i = 0; i < count; i++) {
            try {
                // Spawning processes directly
                Runtime.getRuntime().exec("java -jar myapp.jar");
            } catch (IOException e) {
                System.out.println("Execution failed: " + e.getMessage());
            }
        }
        // If a process dies, it remains dead forever!
    }
}
```

### Better implementation
*A basic thread watchdog that restarts a dead process, but cannot scale dynamically, lacks resource capacity awareness, and cannot schedule containers across multiple nodes.*

```java
import java.util.concurrent.ConcurrentHashMap;

// BETTER: Basic process restarter
public class watchdogService {
    private final ConcurrentHashMap<String, Thread> monitoredTasks = new ConcurrentHashMap<>();

    public void monitorTask(String taskId, Runnable task) {
        Thread worker = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                try {
                    task.run();
                } catch (Exception e) {
                    System.out.println("Task failed, restarting: " + taskId);
                }
            }
        });
        monitoredTasks.put(taskId, worker);
        worker.start();
    }
}
```

### Best implementation
*A simulation of the Kubernetes Control Plane, demonstrating etcd state representation, Scheduler node capacity checks, and the Controller Manager's reconciliation loop.*

```java
import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

// BEST: Kubernetes Controller Manager & Scheduler Simulation
public class MockKubernetesCluster {

    // Pod representation with CPU requirements and health flags
    public static class PodSpec {
        public final String name;
        public final int cpuRequest;
        public boolean isHealthy = true;

        public PodSpec(String name, int cpuRequest) {
            this.name = name;
            this.cpuRequest = cpuRequest;
        }
    }

    // Node representation with capacity limits
    public static class WorkerNode {
        public final String nodeId;
        public final int maxCpuCapacity;
        public int allocatedCpu = 0;
        public final List<PodSpec> scheduledPods = new CopyOnWriteArrayList<>();

        public WorkerNode(String nodeId, int maxCpuCapacity) {
            this.nodeId = nodeId;
            this.maxCpuCapacity = maxCpuCapacity;
        }

        public boolean canHost(PodSpec pod) {
            return (allocatedCpu + pod.cpuRequest) <= maxCpuCapacity;
        }

        public void schedulePod(PodSpec pod) {
            scheduledPods.add(pod);
            allocatedCpu += pod.cpuRequest;
            System.out.println("Scheduled Pod [" + pod.name + "] on Node [" + nodeId + "]");
        }

        public void removePod(PodSpec pod) {
            if (scheduledPods.remove(pod)) {
                allocatedCpu -= pod.cpuRequest;
                System.out.println("Removed Pod [" + pod.name + "] from Node [" + nodeId + "]");
            }
        }
    }

    // Declarative state stored in etcd
    public static class DeploymentSpec {
        public final String name;
        public final int desiredReplicas;
        public final int cpuRequestPerPod;

        public DeploymentSpec(String name, int desiredReplicas, int cpuRequestPerPod) {
            this.name = name;
            this.desiredReplicas = desiredReplicas;
            this.cpuRequestPerPod = cpuRequestPerPod;
        }
    }

    // Control Plane Engine
    public static class ControlPlane {
        private final List<WorkerNode> nodes = new ArrayList<>();
        private DeploymentSpec activeDeployment;
        private final List<PodSpec> allActivePods = new CopyOnWriteArrayList<>();
        private int podCounter = 0;

        public void registerNode(WorkerNode node) {
            nodes.add(node);
        }

        public void setDesiredState(DeploymentSpec spec) {
            this.activeDeployment = spec;
        }

        // 1. Scheduler Component
        private boolean schedulePod(PodSpec pod) {
            for (WorkerNode node : nodes) {
                if (node.canHost(pod)) {
                    node.schedulePod(pod);
                    allActivePods.add(pod);
                    return true;
                }
            }
            System.out.println("Scheduling failed: Insufficient resources in cluster for " + pod.name);
            return false;
        }

        // 2. Controller Manager: The Reconciliation Loop
        public void reconcile() {
            if (activeDeployment == null) return;

            // Remove dead pods first (Simulating Kubelet health status updates)
            for (WorkerNode node : nodes) {
                for (PodSpec pod : node.scheduledPods) {
                    if (!pod.isHealthy) {
                        System.out.println("Reconciler: Found dead pod: " + pod.name);
                        node.removePod(pod);
                        allActivePods.remove(pod);
                    }
                }
            }

            int currentReplicas = allActivePods.size();
            int desiredReplicas = activeDeployment.desiredReplicas;

            System.out.println("Reconciliation Loop: Desired=" + desiredReplicas + ", Actual=" + currentReplicas);

            // If actual < desired, spawn new pods
            if (currentReplicas < desiredReplicas) {
                int countToCreate = desiredReplicas - currentReplicas;
                System.out.println("Reconciler: Creating " + countToCreate + " new pods.");
                for (int i = 0; i < countToCreate; i++) {
                    PodSpec newPod = new PodSpec(activeDeployment.name + "-pod-" + (++podCounter), activeDeployment.cpuRequestPerPod);
                    schedulePod(newPod);
                }
            } 
            // If actual > desired, terminate excess pods
            else if (currentReplicas > desiredReplicas) {
                int countToTerminate = currentReplicas - desiredReplicas;
                System.out.println("Reconciler: Terminating " + countToTerminate + " excess pods.");
                for (int i = 0; i < countToTerminate; i++) {
                    PodSpec excessPod = allActivePods.get(0);
                    for (WorkerNode node : nodes) {
                        node.removePod(excessPod);
                    }
                    allActivePods.remove(excessPod);
                }
            }
        }
    }
}
```

## Step-by-step explanation
1. **Declare Desired State:** A developer submits a YAML manifest defining a Deployment (e.g., v1.0 of API, 3 replicas) to the **API Server**.
2. **Persistence:** The API Server persists the configuration in **etcd**.
3. **Observation (Controller Manager):** The Controller Manager watches etcd and registers that the actual count of running pods (0) does not match the desired count (3).
4. **Scheduling:** The Scheduler scans the worker nodes, evaluates available CPU/Memory resources, selects the best nodes, and binds the Pods to those nodes.
5. **Execution (Kubelet):** The Kubelet agent on the target nodes detects the scheduled Pods and invokes containerd/Docker to spin up the container processes.
6. **Reconciliation (Self-Healing):** If a Pod crashes, the Controller Manager detects the deviation (Actual = 2, Desired = 3) and initiates rescheduling to restore the cluster state.

## Multiple real-world examples
1. **Zero-Downtime Rolling Updates:** Upgrading a web service by gradually spawning new v2 Pods and terminating old v1 Pods only after readiness probes pass.
2. **Horizontal Pod Autoscaling (HPA):** Scaling replica counts up to 20 during traffic surges, and scaling down during off-peak hours.
3. **Internal Service Mesh & DNS:** Pods communicating using predictable hostnames (e.g., `http://payment-service.default.svc.cluster.local`) load-balanced dynamically.
4. **DaemonSets for Logging/Monitoring:** Running exactly one instance of a logging daemon (e.g., Fluentd) on every node in the cluster automatically.
5. **StatefulSet Databases:** Deploying clustered databases (like Cassandra or PostgreSQL) with persistent volumes and fixed network ordinal addresses (`db-0`, `db-1`).

## Pros
- **Self-Healing:** Auto-restarts failed containers and replaces dead nodes.
- **High Availability & Scalability:** Effortlessly scales applications out and in.
- **Infrastructure as Code:** Everything is defined declaratively in YAML, allowing infrastructure to be version-controlled in Git (GitOps).
- **Cloud Agnostic:** You can run K8s on AWS (EKS), GCP (GKE), Azure (AKS), or bare metal data centers.

## Cons
- **Steep Learning Curve:** The ecosystem is incredibly complex with dozens of interacting components and specialized vocabulary.
- **Overhead:** Running the Control Plane consumes significant CPU and RAM. It is overkill for simple, small-scale applications.
- **Operational Complexity:** Managing and upgrading a K8s cluster yourself is notoriously difficult (which is why managed services like EKS/GKE are popular).

## Interview questions

### Beginner
- **Q: What is a Pod in Kubernetes?**
  - **A:** A Pod is the smallest deployable unit in Kubernetes. It usually contains a single container, but can contain multiple tightly coupled containers (like a main app container and a logging sidecar) that share the same IP address and storage.
- **Q: What is the purpose of kubelet?**
  - **A:** Kubelet is the node agent that runs on every worker node in the cluster. It ensures that the containers described in PodSpecs are running and healthy.

### Intermediate
- **Q: What is the difference between a Deployment and a Service?**
  - **A:** A Deployment ensures your application is running (it manages the pods, replicas, and updates). A Service provides a stable, persistent IP address and DNS name to access those pods, load-balancing network traffic across them. Without a Service, you wouldn't be able to reliably route traffic to pods, because pods are frequently killed and reborn with new IP addresses.
- **Q: What are Liveness and Readiness probes?**
  - **A:** 
    - **Readiness Probe:** Checks if a container is ready to accept traffic. If it fails, K8s stops routing traffic to the Pod.
    - **Liveness Probe:** Checks if the container is still running. If it fails, K8s kills the container and restarts it according to its restart policy.

### Senior
- **Q: How does Kubernetes handle State and Storage for databases?**
  - **A:** Containers are ephemeral. To run stateful applications like databases, K8s uses `PersistentVolumes` (PV) and `PersistentVolumeClaims` (PVC). To ensure stable network identities and ordered deployment (e.g., spinning up a Master DB before a Replica DB), K8s uses a `StatefulSet` object instead of a standard `Deployment`.
- **Q: Explain what happens when a node goes offline. How does K8s rebalance the cluster?**
  - **A:** 
    1. The Control Plane API Server stops receiving heartbeats from the node's Kubelet.
    2. After a grace period (default 40s), the node status changes to `NotReady`.
    3. The Controller Manager notices that pods scheduled on this node are unreachable.
    4. If the state persists beyond the eviction timeout (default 5m), the pods are marked for termination.
    5. The Controller Manager creates replacement pod specs in etcd.
    6. The Scheduler assigns these new pods to healthy nodes with available capacity.

### Staff Engineer
- **Q: How does Kubernetes internal networking work? Contrast Calico (IP-in-IP) vs AWS-VPC CNI, and explain how Kube-Proxy implements Services.**
  - **A:** 
    1. **Networking Model:** K8s enforces that every Pod gets a unique, routable IP address in the cluster, and Pods can talk to each other without NAT.
    2. **CNI Plugins:**
       - **Calico:** Uses an overlay network (e.g. VXLAN or IP-in-IP encapsulation) or configures direct BGP routing. This creates an isolated virtual network on top of the host network.
       - **AWS-VPC CNI:** Allocates native AWS Elastic Network Interfaces (ENIs) and secondary IP addresses directly to Pods. This means Pods are native citizens in the VPC network, eliminating encapsulation overhead.
    3. **Kube-Proxy:** Operates on every node to intercept traffic sent to ClusterIP Services.
       - **IPTables Mode (Default):** Kube-Proxy writes netfilter rules in the Linux kernel to perform DNAT, randomly picking a backend Pod IP. It is slow to update on clusters with thousands of services.
       - **IPVS Mode:** Uses IPVS hash tables inside the kernel, providing $O(1)$ routing lookup performance, making it highly scalable for massive clusters.

## Common mistakes
- **Not setting Resource Limits:** If you don't define CPU and Memory requests/limits in your Pod definitions, a single leaky application can consume 100% of a node's resources, starving and crashing other critical pods.
- **Using 'latest' image tags:** If a pod crashes and restarts, it will pull the new 'latest' image. This means your production environment might accidentally upgrade itself to a breaking version without you knowing. Always use specific semantic versions (e.g., `image:v1.2.4`).
- **Ignoring Graceful Shutdown:** Failing to handle the `SIGTERM` signal in container applications, causing immediate process termination that drops active user requests.

## Best practices
- Use Liveness and Readiness Probes.
- Use Namespaces to logically separate environments.
- Enforce Pod Disruption Budgets (PDB) to ensure high availability during node upgrades.

## When NOT to use
- For simple monolithic applications.
- If you don't have a dedicated DevOps/SRE team. Serverless architectures (like AWS Lambda or Fargate) might be easier and cheaper for smaller teams.

## Comparison with similar concepts
- **Kubernetes vs Docker Compose:** Docker Compose is for running multi-container apps on a *single* local machine. K8s is for orchestrating containers across a *cluster* of production machines.
- **Kubernetes vs AWS ECS:** ECS is Amazon's proprietary container orchestrator. It is simpler than K8s but locks you into the AWS ecosystem.

## Summary
Kubernetes is the undisputed king of container orchestration. While it introduces significant complexity, its ability to provide self-healing, auto-scaling, and declarative infrastructure makes it the ultimate platform for running modern, large-scale, cloud-native microservices.
